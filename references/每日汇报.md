# 每日汇报

本参考用于配置 OpenClaw Agent 每日运行 `scripts/daily_report.sh`，汇总当天需要关注的订阅状态，并在发现网络资源高于网盘最高集时询问用户是否更新。

## 使用场景

用户说：

- “帮我每天汇报订阅情况”
- “每天检查今天该更新的剧”
- “发现有新资源时问我要不要更新”
- “帮我配置每日懒人汇报”

Agent 应读取本参考。

## 脚本说明

脚本位置：

```bash
<skills-dir>/quark-lazy-cli/scripts/daily_report.sh
```

用法：

```bash
bash <skills-dir>/quark-lazy-cli/scripts/daily_report.sh [星期几] [env路径]
```

示例：

```bash
bash <skills-dir>/quark-lazy-cli/scripts/daily_report.sh 6 <skills-dir>/quark-lazy-cli/.env
```

参数：

- `星期几`：可选。`1=周一`，`2=周二`，...，`7=周日`。不传则自动使用当天。
- `env路径`：可选。推荐传入 `<skills-dir>/quark-lazy-cli/.env`。

使用前，需要把脚本里的 `剧名1`、`剧名2`、`剧名3` 替换成用户真实的 QAS 任务名。

脚本里的剧集分组应根据 [订阅时间估算.md](订阅时间估算.md) 中维护的“动漫订阅更新时间建议表”设计：

- 周一汇报周一需要关注的剧。
- 周二汇报周二需要关注的剧。
- 如果某部剧有兜底重试时间，也可以同时放入兜底日期。
- `run_status "剧名1"` 里的剧名必须使用表格里的 `taskname`，不要使用随意昵称。

示例：

```md
| taskname | 剧名/显示名 | 更新策略 | 建议订阅更新时间 | 定时表达式 | 状态 | 备注 |
|----------|-------------|----------|------------------|-------------|------|------|
| 剧名1 | 剧名1 | 画质优先 | 周一 22:00 | `0 22 * * 1` | 正常 | |
| 剧名2 | 剧名2 | 画质优先 | 周一 23:00 | `0 23 * * 1` | 正常 | |
| 剧名3 | 剧名3 | 时间优先 | 周二 20:30 | `30 20 * * 2` | 正常 | |
```

对应脚本：

```bash
1)
  echo "📅 周一"
  run_status "剧名1"
  run_status "剧名2"
  ;;
2)
  echo "📅 周二"
  run_status "剧名3"
  ;;
```

## OpenClaw 定时任务模板

```bash
openclaw cron add \
  --name "qslazy-每日订阅汇报" \
  --cron "30 23 * * *" \
  --tz "Asia/Shanghai" \
  --session isolated \
  --agent AgentName \
  --announce \
  --channel feishu \
  --account bot-AgentName \
  --to "USER_OPEN_ID" \
  --message "bash -c 'bash <skills-dir>/quark-lazy-cli/scripts/daily_report.sh \"\" <skills-dir>/quark-lazy-cli/.env'\n\n请根据 stdout 汇报订阅状态。如果发现网络资源的最高集数高于网盘最高集，先询问用户是否执行 qslazy update all，不要自动更新。"
```

替换项：

- `AgentName`：执行汇报的 OpenClaw Agent。
- `bot-AgentName`：通知账号。
- `USER_OPEN_ID`：通知目标。
- `<skills-dir>/quark-lazy-cli`：实际 Skill 安装目录。

## Agent 汇报规则

Agent 只根据 stdout 汇报，除非排错，不读取 log 或报告文件。

每个任务建议汇报：

```text
[ 凡人修仙传 ]

💾 网盘：最高 82（更新于：昨天），最低 78，缺[79, 80]
🎯 目标：第82集以上资源
💎 偏好：最高画质优先

🔍 最新资源：今天和昨天
🔍 资源更新至：83集

状态总结：网盘最高 82 vs 资源更新至 83，需要更新；同时网盘缺[79, 80]，建议补漏。
```

状态文案建议：

```text
状态总结：网盘最高 X vs 资源更新至 X，网盘已是最新。
状态总结：网盘最高 X vs 资源更新至 Y，需要更新。
状态总结：网盘最高 X vs 资源更新至 Y，需要更新；同时网盘缺[...]，建议补漏。
状态总结：资源编号或版本较复杂，需要人工确认。
```

## 发现新资源时的处理

如果 stdout 显示网络候选资源高于网盘最高集，例如：

```text
网盘：最高 22
候选资源：更新至 23 集
```

Agent 不能自动更新，必须先按上面的格式汇报，再问用户：

```text
我看到《剧名1》网盘最高到第 22 集，但网络资源已更新到第 23 集。
是否现在执行 `qslazy update all 剧名1 --env <skills-dir>/quark-lazy-cli/.env` 追新并补漏？
```

用户确认后再执行：

```bash
LAZY_CLI_ADVISOR=code qslazy update all 剧名1 --env <skills-dir>/quark-lazy-cli/.env
```

如果资源命名复杂、编号冲突或画质选择困难，改用：

```bash
LAZY_CLI_ADVISOR=llm qslazy update all 剧名1 \
  --env <skills-dir>/quark-lazy-cli/.env \
  --add-prompt "写入该剧的特殊筛选规则"
```

## 关键限制

- 每日汇报是只读检查，默认不更新订阅。
- 用户未授权时，不执行 `update new` 或 `update all`。
- 发现网络资源更高时，只能询问用户是否更新。
- 不要把真实 QAS Token、Feishu open_id 或用户私有路径写入公开模板。
